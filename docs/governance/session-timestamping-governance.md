# Session Timestamping Governance

## Purpose
Session timestamping provides chronological clarity, traceability, and alignment across all project sessions.  
It anchors discussions, prevents drift, and supports professional team workflows.

## Timestamp Format
Use the canonical format:

**YYYYMMDD + descriptor**

Examples:
- **20260706 evening session**
- **20260707 morning session**
- **20260707 architecture session**

This format is:
- Chronologically sortable  
- Human‑readable  
- Machine‑friendly  
- Consistent across all documents  

## Placement Rules
Timestamp MUST appear at the start of every session, in this order:

1. **Timestamp**
2. **Current Kanban Status**
3. **Short Professional Briefing**
4. **Session Execution**
5. **Professional Team Summary**

Timestamping is mandatory and applies to all sessions without exception.

## Session Structure
### 1. Timestamp
Marks the session anchor point.

### 2. Current Kanban Status
Full, authoritative, flat Kanban board rendered exactly as stored.

### 3. Short Professional Briefing
Includes:
- Where we are in the progress  
- What was done yesterday / last session  
- What is planned for today  
- Observations, blockers, questions  

### 4. Session Execution
Work performed according to Priority list and governance rules.

### 5. Professional Team Summary
End‑of‑session wrap‑up similar to a real engineering team meeting.

## Benefits
- Chronological traceability  
- Clear separation between sessions  
- Audit‑friendly documentation  
- Easy reference for past discussions  
- Eliminates mismatched assumptions  
- Supports toggling between topics across days  

## Lessons Learned Integration
Every lesson learned MUST reference:
- Session timestamp  
- Session descriptor  
- Context of discovery  
- Trigger  
- Resolution  
- Impact  

This ensures a fully traceable Lessons Learned repository.

## Governance Alignment
Timestamping is now a mandatory part of:
- Project governance  
- Kanban workflow  
- Documentation model  
- Lessons Learned  
- Architecture decision records  
- Learning area documents  

No deviations.  
No assumptions when unclear.  
No silly actions.
